LLM 是什么？一个直觉


大语言模型 ≈ 一个读过人类大量文本的「统计语感」


它做不了的事

  • 不是数据库，不「存储」事实
  • 不是搜索引擎，不「检索」信息
  • 数学计算经常出错

它拿手的事

  • 理解和生成自然语言
  • 模式识别、类比推理
  • 根据上下文「补全」内容
不讲Transformer细节。强调LLM的核心是「语感」而非「知识」。

2026年的 AI 工具全景

不是一个工具，是一整个生态

模型层

ChatGPT (OpenAI) · DeepSeek · Claude (Anthropic) · Gemini (Google) · Llama (Meta) · Qwen (阿里) · ...

竞争激烈，能力趋同，选哪个不是关键

同一个模型，用法不同，产出天差地别

模型只是基础设施。你怎么用它，才决定了你的生产力。
不做品牌对比。强调生态思维：模型能力趋同，差异在应用层。

为什么强调「生态」？


2024年大家问的是：「哪个模型最好？」
2025年变成了：「哪个工具链最好？」
到了2026年，问题变成了：「我的研究基础设施该怎么搭？」


模型会不断更新换代。DeepSeek-V3 出来了，Claude 4.5 出来了，GPT-5 出来了……

但你搭的工作流、写的配置文件、积累的方法论知识，换个模型照样能用。这些才值得花时间。

这是整场讲座的核心论点之一：投资工作流而非单一工具。

社科研究者关心什么？


文本与语言

话语分析、编码、开放题处理、文献综述……社科研究里到处都是文本数据，LLM 在这方面确实好用

因果推断

选什么方法、怎么诊断模型、怎么解释结果？这些地方需要方法论上的硬约束

理论构建

讨论机制、跟文献对话、辨析概念……你可以拿 AI 当一个可以随时 argue 的同事


社科跟工程不一样。代码跑通了不算完，因果识别对不对、理论机制说不说得通，这些才是我们真正在意的。
社科的三个核心需求区别于个人开发者和自然科学研究者。

方法

三个核心组件

AGENTS.md · Skills · MCP

先说清楚：什么是 Agent？

Agent = 能自主规划、使用工具、持续执行任务的 AI 系统

关键区别：它不只是回答问题，而是自己决定下一步做什么 — 能读文件、调工具、跑代码

Coding Agent

Cursor Claude Code Codex Windsurf

能读你的代码库、自动改代码、跑测试。这些产品本质上做的是一回事，只是界面和体验不同。

Research Agent

ChatGPT Deep Research

自主搜索数十个网页、综合分析、生成研究报告。跟 Coding Agent 不同，面向的是信息检索和综合。

你可能已经在用 Agent 了 — 如果你用过 Cursor，它就是一个 Coding Agent。只是你可能没意识到它跟 ChatGPT 聊天的本质区别。
先给Agent下定义，再分两类产品举例。让听众意识到Agent不是新概念，他们可能已经在用了。

从聊天到 Agent：什么变了？


聊天模式

  • 一次性对话
  • 没有项目记忆
  • 碰不到文件和工具
  • 你说什么它做什么
→

Agent 模式

  • 持续的项目上下文
  • 能读你的代码和数据
  • 调用工具、运行代码
  • 主动规划和执行

Agent = 模型 + 项目记忆 (AGENTS.md) + 领域知识 (Skills) + 工具连接 (MCP)

这是理解后续内容的关键框架。Agent不是更好的聊天，是不同的范式。

AGENTS.md：给 AI 的项目说明书

类比：实验室给新 RA 的入职手册

你会给新来的研究助理什么？

  • 项目的基本规则
  • 前人踩过的坑
  • 代码和数据的组织惯例
  • 绝对不能做的事

AGENTS.md 做的是一样的事：

一个放在项目根目录的 Markdown 文件，告诉 AI：

  • 项目结构和惯例
  • 已知的陷阱和注意事项
  • 测试和构建命令
  • 代码风格要求

AGENTS.md 已被 60,000+ 开源项目采用，由 Linux Foundation 管理 — AGENTS.md

AGENTS.md是业界标准。不同AI工具都支持读取这个文件。

AGENTS.md 写什么、不写什么

少即是多，精准才有用

写进去

  • AI 猜不到的命令（特殊构建步骤、非标准测试入口）
  • 项目特有的坑（这个列名有歧义、那个函数已弃用）
  • 测试和验证命令（怎么跑测试、怎么检查结果）
  • 偏离常规的风格（我们用 tidyverse 不用 data.table）

别写

  • AI 读代码就能看出的（函数签名、目录结构）
  • 通用常识（"用 git 做版本控制"）
  • 大段理论背景（放进 Skill 或单独文档）
  • 频繁变动的信息（每周都变的 TODO）
像维护代码一样维护 AGENTS.md — 文件太臃肿，AI 反而忽略重点。
AGENTS.md 不是越长越好。保持精炼，AI 才能聚焦关键信息。

AGENTS.md 的分层与跨工具支持

一份配置，全生态通用

三级目录层次


~/.claude/AGENTS.md        ← 全局（所有项目）
project/AGENTS.md          ← 项目（当前项目）
project/src/AGENTS.md      ← 子目录（局部覆盖）

就近优先：子目录的规则覆盖项目级别

引用语法


@docs/THEORY.md
@regression-tables.md

用 @path 引入外部文档，避免重复

跨工具生态支持

以下工具都读取 AGENTS.md：

Claude Code GitHub Copilot Cursor OpenAI Codex Windsurf Cline Aider 20+ more

Linux Foundation 管理的开放标准，不被任何厂商锁定

写一份，所有 AI 工具都能读。这就是投资开放标准的价值。
分层设计让你可以全局设置风格偏好，项目级别设置具体规则。跨工具支持意味着不怕工具切换。

Agent Skills：给 AI 的领域知识包

AGENTS.md 解决了项目上下文，Skills 再给 AI 加上领域专长

什么是 Skill？

一个文件夹，里面包含指令、知识、脚本，让 AI 获得特定领域的专业能力。

方法论

因果推断方法选择、多重插补诊断、面板数据分析...

流程

论文审稿标准、回归表格规范、文献综述框架...

自动化

代码审计、可视化检查、版本控制流程...

Agent Skills 是开放标准，已被 20+ AI 工具支持 — agentskills.io

Skills是可复用的。写一次，在不同AI工具中都能使用。

Skill 的规范格式与渐进加载

结构化的知识，按需释放 — 颜色对应加载阶段

--- name: causal-inference ← 阶段 1 description: | 因果推断方法选择与诊断。 Use when: DID, 面板数据因果推断 --- my-skill/ ├── SKILL.md ← 阶段 2 ├── references/ ← 阶段 3 │ └── diagnostics.md └── scripts/ └── did_template.R
1 启动：~100 tokens
名称 + 描述 + 触发词 — AI 扫一眼就知道该不该加载
2 激活：<5000 tokens
SKILL.md 正文 — 核心指令 + 决策树
3 按需：不限
templates/ + examples/ — 只在需要时加载
左侧代码的颜色和右侧阶段卡片一一对应。渐进加载的核心是节省上下文窗口，只在需要时释放知识。

一个因果推断 Skill 长什么样？

这就是一个完整的 SKILL.md 正文部分


# Causal Inference Method Selection

## Decision Tree
Parallel trends hold?
├─ Yes → Two-way Fixed Effects (TWFE)
└─ No  → Factor structure expected?
         ├─ Yes → Interactive Fixed Effects (IFE)
         └─ No  → Matrix Completion (MC)

## Diagnostic Checklist
1. Run placebo test (pre-treatment effects)
2. Visual inspection (gap plot, equivalence plot)
3. Check pre-trends significance
4. Sensitivity analysis (breakdown frontier)

## Common Pitfalls
- Don't use TWFE with staggered treatment
- Always report pre-treatment fit statistics
- Bootstrap SE needs ≥200 iterations for stability

虚构的通用示例，真实 Skill 会包含更多参数细节和代码模板

关键：把方法论教科书里的决策树编码成AI可用的格式。这不需要CS背景。

MCP：AI 通往外部世界的接口

有了项目知识和领域专长，还差动手能力

类比：USB-C 让你的电脑连接各种设备。MCP 让 AI 连接各种工具和数据源。


有了 MCP，AI 就不只是聊天了，它可以直接读你的数据、跑你的分析、管理你的文件。

MCP 是开放协议 — modelcontextprotocol.io

MCP让AI从「聊天助手」变成「有手有脚的研究助理」。下一张详细讲三个核心概念。

MCP 架构：三个核心概念

不同的控制权，不同的用途

Tools

AI 控制

AI 主动调用的操作：跑回归、清洗数据、生成图表

社科示例：AI 自动选择 DID/SCM 并执行

Resources

应用控制

AI 可以读取的数据：面板数据、文献库、调查问卷

社科示例：AI 读取 Zotero 文献元数据

Prompts

用户控制

用户触发的预设流程：审稿模板、分析报告生成

社科示例：一键生成论文诊断报告

Host（IDE / 终端）→ Client（AI 应用）→ Server（你的工具和数据）
Tools/Resources/Prompts 三种 primitive 对应三种控制模式。Host-Client-Server 是 MCP 的通信链路。

一个 MCP Server 有多简单？

Python + FastMCP，十几行就够 — 颜色对应右侧概念

from fastmcp import FastMCP mcp = FastMCP("社科研究助手") @mcp.tool() ← Tool：AI 主动调用 def run_did(data_path: str, treat_col: str, time_col: str) -> str: """执行双重差分分析""" # ... 分析逻辑 ... return f"ATT = {att:.3f}" @mcp.resource("data://codebook") ← Resource：AI 读取 def get_codebook() -> str: return open("codebook.md").read()

@mcp.tool()

把一个普通 Python 函数变成 AI 可调用的工具。AI 看到函数名和 docstring，自己决定何时调用。

@mcp.resource()

把数据暴露给 AI 读取。AI 需要上下文时自动获取，不需要你手动粘贴。

已有的分析代码加两行装饰器，就能变成 AI 可调用的工具。不需要学新框架。
FastMCP 是官方推荐的 Python SDK。颜色对应：绿色=Tool（AI主动调用），蓝色=Resource（AI读取数据）。

实践 1：AI 辅助大规模文本分析

场景

一个涉及大量中文学术文本的研究项目，需要对文本进行多维度分类和测量。

AI 参与了什么？

  • 搭建 BERT 分类器的代码框架和训练流水线
  • 调试嵌入空间 (Embedding) 的维度分析
  • 优化并行处理架构，提升百万级数据的处理效率
一个意外收获：嵌入空间里的「异常」结果其实不是 AI 出了错，而是数据本身的结构就是那样的。
不透露具体项目。重点是AI帮助发现数据结构的故事。

实践 2：LLM 做信息抽取的真实教训

场景

用 LLM 从非结构化网页中抽取结构化信息（人名、职称、机构等）。

第1次：直接写 prompt，看起来不错

实际结果：LLM 把名字和机构名拼在一起、把「助理教授」匹配成「教授」...

→

第8次：处理了所有 edge cases

关键：靠的不是更好的 prompt，而是更好的后处理逻辑

LLM 不是开箱即用的。从「看起来能用」到「真正可靠」，中间隔了系统性测试和大量迭代。
这个故事很有代表性。很多人在第1次就认为「AI搞定了」，但真正的工作在后面。

实践 3：AI 作为「思考的对话者」

场景

一个调查实验项目，某个处理条件的结果不符合预期。传统做法：是不是代码写错了？样本量不够？

和 AI 的对话过程：

  1. AI：你的处理效应为什么包含间接效应？试试分解看看
  2. 我：控制了中介变量后，直接效应反转了
  3. AI：这是经典的抑制效应 (suppression effect)，可能恰恰是你论文的亮点
这个例子说明的是：AI 在社科里的用处，不只是帮你跑回归。它还能帮你想清楚为什么结果跟理论对不上。
这里体现了社科的特殊性：理论驱动。AI帮你思考，不只是帮你执行。

「Let It Crash」原则

为什么学术代码不能静默降级

静默降级

try:
    score = model.predict(text)
except:
    score = 0  # 悄悄赋默认值

研究者不知道哪些数据用了降级值。论文发表后才发现。

vs

显式失败

try:
    score = model.predict(text)
except Exception as e:
    logger.error(f"Failed: {e}")
    raise  # 立刻停止

出错立刻知道。宁可中断流水线，也别带着隐患继续跑。

学术代码跟产品代码不一样。我们追求的不是「不崩溃」，而是「每个数字都可溯源」。
这个原则对社科研究者特别重要。你的论文中的每一个系数都必须是透明的。

AGENTS.md 作为「制度记忆」


每一条规则背后都是一个真实的错误。

把它写进 AGENTS.md，这个错误就不会再犯第二次。



## Known Pitfalls

| Issue | Solution |
|-------|----------|
| Merge by position fails silently | Always use explicit ID keys |
| Function was renamed in v2 | Use new_name(), not old_name() |
| Cache ignores data changes | Include data fingerprint in cache key |
| Silent fallback hides errors | Remove all try-except with defaults |

这就是所谓的「研究工程化」：把个人经验变成系统化的规则。

和写实验日志一个道理。但这次是写给AI看的「实验日志」。

AI 的输出是假说，不是结论

容易踩的坑

  • 「AI 写的代码跑通了，应该没问题吧」
  • 「结果和预期差不多，直接用了」
  • 「AI 说这个方法合适，那就用这个」

应该怎么做

  • 每次修改后比对前后统计量
  • 先拿小数据集验证逻辑，再跑全量
  • AI 推荐的方法，回去查原始文献
我在 AGENTS.md 里写的第一条规则就是：
"Verify outputs after code changes — compare before/after statistics."
这不是不信任AI，而是学术研究的基本功。AI让你快了，但不能让你放松验证。

Skill 1: latex-tables

回归表格规范 · 标准层

github.com/SiyaoZheng/ai4ss-skills/blob/main/latex-tables.skill

解决什么问题？

每次做回归表格都要查期刊格式要求，星号放哪、标准误怎么写、用几位小数……

AI 会做

  • booktabs 三线表
  • 显著性标记 + 标准误注释
  • AEA / APSR 期刊格式

触发方式

「把这个回归结果转成 LaTeX 表格」

「按 AEA 格式做一个 summary statistics」

最简单的一个skill，解决的是格式标准化问题。

Skill 2: analysis-explainer

分析结果文档化 · 流程层

github.com/SiyaoZheng/ai4ss-skills/blob/main/analysis-explainer.skill

解决什么问题？

跑完分析，合作者问「你用的什么方法？参数是多少？」每次都要重新整理一遍。

AI 会做

  • 自动嵌入所有图表
  • 记录方法参数（seed、迭代次数等）
  • 生成 PDF 技术文档

触发方式

「帮我解释这些分析结果」

「写一份技术说明给合作者」

内置 style-guide：避免 AI 腔，输出符合学术写作规范

这个skill的价值在于让分析过程可复现、可沟通。

Skill 3: r-performance

R 代码性能优化 · 工程层

github.com/SiyaoZheng/ai4ss-skills/blob/main/r-performance.skill

解决什么问题？

代码跑得慢，但不知道慢在哪、怎么优化。想并行但不会配置 HPC。

AI 会做

  • 用 profvis 定位瓶颈
  • 向量化 / 预分配 / 并行化决策
  • HPC 集群配置（Slurm/SGE/PBS）

触发方式

「这段代码跑得太慢了」

「如何在 HPC 上并行运行」

包含 5 个参考文档：profiling、r-internals、future 生态、HPC 配置、常见模式

这个skill内容最多，覆盖从本地优化到集群并行的完整链路。
